## Session Work

### Bug Fix 1: RE-45 showing as "Unknown" weapon
- **Root cause**: GEP sends weapon slot keys in inconsistent formats: `weapon_0` (underscore), `0` (bare numeric), or `weapon0` (canonical). The weapons handler stored whatever keys GEP sent without normalization. The kill event weapon fallback looked for `weapon0` or `0` but missed `weapon_0`. The renderer WeaponTracker also only checked `weapon0` and `0`.
- **Fix**: Added `normalizeWeaponSlotKey()` method to EventProcessor that maps:
  - `weapon_0` -> `weapon0` (ow-electron underscore variant)
  - `0` -> `weapon0` (bare numeric variant)
  - `weapon0` -> `weapon0` (already canonical, pass through)
  - `inUse` -> `inUse` (non-slot keys pass through)
- **Renderer fix**: Added `weapon_0` fallback in WeaponTracker.tsx as defense-in-depth
- **Logging**: Added WEAPON-DEBUG logging for normalization events and kill fallback usage
- **Tests**: 6 new tests covering all normalization variants + kill event fallback
- **Files changed**:
  - `src/main/gep/event-processor.ts` - normalizeWeaponSlotKey + weapons handler + kill fallback logging
  - `src/renderer/windows/MainOverlay/WeaponTracker.tsx` - weapon_0 fallback
  - `src/main/gep/__tests__/event-processor.test.ts` - 6 new normalization tests

### Bug Fix 3 (continued): RE-45 STILL showing "Unknown" -- weapon NAME cleaning
- **Root cause analysis**: The prior fix (normalizeWeaponSlotKey) handled slot KEY normalization correctly. But the weapon NAME value itself might arrive as a localization key (`#weapon_re45_auto`) or internal engine name (`weapon_re45_auto`) instead of clean display name.
- **Fix**: Added `cleanWeaponName()` utility in `src/shared/utils.ts`:
  - Strips `#weapon_` localization prefix (case-insensitive)
  - Strips `weapon_` internal engine prefix (case-insensitive)
  - Strips `_NAME` suffix if present
  - Maps 40+ known internal weapon names to display names (e.g., `re45` -> `RE-45 Auto`, `flatline` -> `VK-47 Flatline`)
  - Passes through already-clean display names unchanged
  - Returns "Unknown" for null/empty
- **Applied**: `cleanWeaponName()` called on every weapon name value in the weapons handler
- **Debug logging**: Enhanced WEAPON-DEBUG logging to show both raw and cleaned weapon names
- **Tests**: 14 new cleanWeaponName tests in `tests/shared/utils.test.ts` + 5 new weapon name cleaning tests in event processor tests

### Bug Fix 4: Rank bar completely missing
- **Root cause**: The `rank` GEP feature ONLY sends `victory` (true/false). It does NOT send rank name or score. The RankedProgress component reads `rankName` and `rankScore` from match store, but no code was producing these values.
- **Research confirmed**: Overwolf official docs show the rank feature only has `victory` key. There is no `rank_info`, `rankScore`, or `rankName` GEP info update.
- **Fix (two-pronged)**:
  1. **API pipeline**: The mozambiquehe.re API player profile already has `rankName` and `rankScore`. Added `onPlayerProfile()` callback to ApiScheduler. In `index.ts`, when profile arrives, broadcast `IPC.MATCH_UPDATE` with `type: 'rank'` data so the existing match store handler picks it up.
  2. **Renderer pipeline**: SessionTracker now also listens for `IPC.API_PLAYER_PROFILE` and extracts rank data from API profiles as defense-in-depth.
  3. **Defensive GEP handlers**: Added handlers in EventProcessor for hypothetical rank keys (`rank_info`, `rank_score`, `current_rank`, `rank_tier`, `rank_level`) in case ow-electron ever sends them. Also added catch-all logging for ANY unhandled keys from the `rank` feature.
- **Tests**: 6 new rank info handler tests in event processor tests
- **Files changed**:
  - `src/main/gep/event-processor.ts` - rank info key handlers + rank debug logging
  - `src/main/index.ts` - API profile -> rank update broadcast
  - `src/main/api/api-scheduler.ts` - onPlayerProfile() callback
  - `src/renderer/windows/MainOverlay/SessionTracker.tsx` - API profile rank listener
  - `src/main/gep/__tests__/event-processor.test.ts` - 6 new rank tests

## Key Learnings
1. GEP weapon slot keys vary between ow-electron versions: `weapon0`, `weapon_0`, `0` are all possible
2. Always normalize incoming GEP data at the handler level, not at each consumer
3. Session-scoped queries need matching database indexes
4. Cross-session queries in coaching rules are intentional for pattern detection, but per-weapon coaching should be session-scoped
5. GEP `rank` feature ONLY sends `victory` (true/false) -- rank name/score must come from external API (mozambiquehe.re)
6. Weapon names from GEP are typically clean display strings, but defense-in-depth cleaning handles localization keys and internal names
7. When an overlay component shows "null" data, trace the FULL pipeline: GEP -> EventProcessor -> domain event -> IPC broadcast -> store handler -> component. The break may be at any level.
8. The mozambiquehe.re API is the authoritative source for rank data, player level, and selected legend (as fallback)
